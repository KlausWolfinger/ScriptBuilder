Detailed Description

Domain Code Definitions and Methods.

Domain Codes allow an administrator to link a code (alphanumeric SIP URI, domain name or partial domain name) with a Destination, an Intercept, an Invalid Code or Fallback to Local Numbering Plan. Configuring Fallback to Local Numbering Plan allows an administrator to purposely fail translation in the common or global numbering plan in order to invoke the fallback destination of a private numbering plan domain code. The Domain Codes are supported by the BG private numbering plans, the BG common numbering plans and the global numbering plan.

  • Domain Codes support limiting applicability of entries based on Routing Areas and Classes of Service.
  • Domain Codes support assigning a prefix type to an entry. Allowed prefix types are: On-net Access, Off-net Access. This distinction is required for features like authorization codes that depend on the prefix type to trigger the feature execution.
  • Domain Codes support assigning a traffic type to an entry. Allowed traffic types are any of the defined traffic types in Global Translation and Routing and None. This allows administrators to activate the Toll Restriction Service to restrict access to certain domains for domain-dialed calls. Authorization Codes, account codes and call forwarding restrictions are supported as well.
  • Domain Codes support delegating the translation from a BG's private numbering plan to the BG's common numbering plan or the global numbering plan.
  • Domain Codes support delegating the translation from a BG's common numbering plan to the global numbering plan.
  • Domain Codes support transitioning an alphanumeric SIP URI to a number and request retranslation of the number (same as Code Processing table).
  • Retranslation shall occur in the Prefix Access Codes table of the calling party's private numbering plan (i.e. only numbers with unknown nature of address can be entered).

Allowed codes within the Domain Code table are:

  • Fully Qualified Domain Names. If the hostport of the URI contains a hostname. (E.g. acme.com in peter.nosp@m.@acm.nosp@m.e.com).
  • Domain names. If the hostport of the URI contains a hostname. The domain name part must match the last domain labels or the top label of the hostname of the URI. The syntax for this type of code is to precede the domain name with the string "*.". (E.g. *.com matches peter.nosp@m.@acm.nosp@m.e.com).
  • IPv4 Address. If the hostport of the URI contains an IPv4Address. (E.g. 192.168.0.15 matches peter.nosp@m.@192.nosp@m..168..nosp@m.0.15). The IPv4 address in the received URI shall be normalized before the verification.
  • IPv6 Address or Reference. If the hostport of the URI contains an IPv6Reference. (E.g. 2001:0db8:85a3::8a2e:0370:7334 matches peter@[2001:0db8:85a3::8a2e:0370:7334]). The IPv6 reference in the received URI shall be normalized before the verification.
  • Alphanumeric SIP URI. The entire alphanumeric SIP URI must match. These codes always contain a userinfo part in addition to the hostport of the URI. The userinfo part always ends on the "@" character. Retranslation is supported for these codes. A special alphanumeric SIP URI is #@hostport which serves to indicate any numeric SIP URI (i.e. a SIP URI for which the userinfo part only contains numeric information (such as '+', '*', '#', 0-9).

Hostnames, IPv4 Addresses or IPv6 Addresses/References of codes entered by the administrator are always normalized prior to storage in shared memory. It is not allowed to enter different formats of the same hostname, IPv4 address, IPv6 address/reference, routing area and class of service combination. Hostname normalization consists of converting the uppercase characters to lowercase characters. All code matches within the domain codes table are therefore case-insensitive. Any of the hostnames, addresses or alphanumeric SIP URIs may be followed by a specific port. If no port is specified then the entry applies to all 65535 ports.
